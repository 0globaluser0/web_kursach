# Структура пояснительной записки

## Обязательная структура

Рекомендуемая структура ПЗ:

1. Титульный лист.
2. Аннотация.
3. Бланк задания, если требуется преподавателем.
4. Содержание.
5. Введение.
6. ГЛАВА 1. Анализ предметной области.
7. ГЛАВА 2. Проектирование базы данных.
8. ГЛАВА 3. Разработка сайта.
9. ГЛАВА 4. Тестирование.
10. Заключение.
11. Список сокращений.
12. Список источников.
13. Приложение А. Описание файлов и текст программы.
14. Приложение Б. Блок-схема.
15. Приложение В. Схема данных, экранные формы и дополнительные материалы.

## Введение

Во введении нужно указать:

- актуальность темы;
- цель курсового проекта;
- задачи проекта;
- объект и предмет разработки;
- краткое описание реализованного сайта;
- используемые технологии.

Для текущего проекта цель можно формулировать так: разработать учебный сайт для симуляции торговли криптовалютой с авторизацией, регистрацией, рынком активов, портфелем, историей сделок и административной панелью.

## Глава 1. Анализ предметной области

Нужно раскрыть:

- что такое криптовалютная биржа;
- какие основные функции есть у торговых платформ;
- какие роли есть в системе: трейдер и администратор;
- какие ограничения введены в учебной модели;
- почему выбран демонстрационный баланс и хранение данных в браузере;
- какие аналоги или типовые функции используются в предметной области.

## Глава 2. Проектирование данных

Нужно описать:

- сущности базы данных;
- поля сущностей;
- связи между сущностями;
- назначение каждой таблицы;
- схему данных как рисунок или таблицу;
- почему сущностей больше пяти;
- как данные в проекте сохраняются через `localStorage`.

## Глава 3. Разработка сайта

Нужно описать:

- структуру файлов проекта;
- назначение каждой HTML-страницы;
- общие стили и темную тему;
- авторизацию и регистрацию;
- рынок активов;
- торговую форму;
- портфель;
- историю операций;
- админ-панель;
- основные функции JavaScript.

## Глава 4. Тестирование

Нужно привести таблицу тестов:

- номер теста;
- проверяемая функция;
- входные данные;
- ожидаемый результат;
- фактический результат;
- статус.

Минимальные тесты:

- вход трейдера;
- вход администратора;
- регистрация нового пользователя;
- ошибка входа;
- покупка при достаточном балансе;
- покупка при недостаточном балансе;
- продажа при наличии монет;
- продажа при нехватке монет;
- просмотр портфеля;
- просмотр истории;
- обновление курса администратором;
- добавление монеты администратором;
- проверка запрета админки для обычного пользователя.

## Заключение

В заключении нужно указать:

- достигнута ли цель;
- какие функции реализованы;
- какие сущности данных спроектированы;
- что проверено тестированием;
- возможные направления развития проекта.
