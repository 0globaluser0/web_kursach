# CryptoTrade

Учебный сайт для симуляции торговли криптовалютой.

## Как запустить

Из корня проекта можно запустить файл:

```text
start_cryptotrade.bat
```

Он откроет локальный сервер и автоматически запустит сайт в браузере.

Запустить локальный сервер:

```powershell
cd C:\shit2\web_kursach\site
python backend.py
```

Открыть в браузере:

```text
http://127.0.0.1:8000/index.html
```

Открывать страницы напрямую как `file:///C:/shit2/web_kursach/site/dashboard.html` не нужно: в таком режиме браузер показывает только HTML-файл и не запускает API. SQLite-база доступна только через локальный сервер `backend.py`, потому что браузер по правилам безопасности не может напрямую читать и изменять файл `cryptotrade.db`.

## Где хранится база данных

Данные хранятся в реальной SQLite-базе `site/cryptotrade.db`. Файл создается автоматически при запуске `backend.py` и не удаляется при закрытии сервера. При следующем запуске сайт продолжает работать с тем же файлом базы. Удалить базу можно только вручную, например через проводник, если удалить `cryptotrade.db`.

SQLite является полноценной файловой СУБД и подходит для курсовой работы: в проекте есть реальные таблицы, внешние ключи, связанные сущности, сохранение пользователей, портфелей, транзакций, истории цен и настроек симуляции.

Для промышленной криптобиржи SQLite недостаточно: там нужна серверная СУБД уровня PostgreSQL/MySQL, хеширование паролей, миграции, резервные копии и серверная авторизация. Для учебного локального проекта SQLite выглядит нормально и защищается проще, чем `localStorage`.

`localStorage` остался только для технических настроек браузера: тема, текущая локальная сессия и служебный статус последнего запроса курса.

## Курсы криптовалют

Предустановленные BTC, ETH, SOL, ADA, BNB, XRP, DOGE, AVAX, DOT, LINK, LTC и TRX могут автоматически обновляться из открытого API Binance Spot по публичным торговым парам. Плановое обновление выполняется раз в 30 секунд, также администратор может запустить обновление вручную.

Администратор управляет режимом цены для каждого актива:

- ручной режим - цена задается администратором и не перезаписывается автообновлением;
- авто из Binance - цена обновляется по указанной биржевой паре, а последняя успешная цена сохраняется в SQLite.

## График и симуляция

На странице торговли есть расширенный график: линия или японские свечи, таймфреймы 1м, 5м, 15м, 1ч и индикатор SMA(5). При запуске через `backend.py` график получает до 240 реальных свечей через `/api/klines`: сначала Binance Spot, затем Coinbase Exchange как запасной источник. Мини-графики на страницах обзора и рынка тоже используют close-цены из этих свечей. Если внешняя сеть недоступна, используется локальная история цен.

В админ-панели есть режим симуляции активности. Администратор вручную задает количество симулируемых пользователей и количество сделок в минуту. Сделки создаются равномерно по времени, а не одной пачкой. Все симулируемые записи получают метку `isSimulated`; при выключении режима удаляются только они, а реальные пользователи и реальные сделки остаются.

Если сервер или сайт были закрыты, симуляция не догоняет пропущенное время после нового запуска. Режим остается включенным, но генерация сделок продолжается только с момента нового открытия сайта.

## Связанные сущности данных

- `roles`
- `users`
- `currencies`
- `wallets`
- `transactions`
- `price_history`
- `system_settings`

`orders` отдельно не используется: в учебной модели сделка исполняется сразу, поэтому запись сразу сохраняется как завершенная транзакция. Таблица `price_history` нужна для внутренней истории последних сохраненных цен: ручных монет, fallback-графиков без внешней сети и отображения локальной динамики курса, если внешний сервис недоступен.

В таблице ролей используются `trader`, `admin` и `simulator`. Роль `simulator` выдается пользователям, созданным режимом симуляции активности.
